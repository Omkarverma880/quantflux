Comprehensive Application Documentation

AutoTrader is a full-stack trading platform that combines a FastAPI backend, a React/Vite frontend, Zerodha Kite Connect integration, automated strategy execution, and a manual options trading console. This document covers the complete application structure, runtime model, core modules, API surface, data flow, and current operational caveats.

Overview

What This Application Does

Primary Capabilities

  • Authenticates with Zerodha and persists the access token for daily reuse.
  • Runs algorithmic strategies from a CLI engine or from web-triggered strategy control routes.
  • Shows dashboard summaries, quotes, positions, holdings, and order history.
  • Exposes a dedicated cumulative-volume analysis workflow for index futures.
  • Provides two strategy-specific trading modules based on Gann levels plus cumulative volume.
  • Provides a manual options trading section with option chain style strike selection for NIFTY and SENSEX.
  • Supports both paper trading and live trading depending on environment flags.

High-Level User Journeys

  • Login to Zerodha, open the dashboard, inspect account and market state, then run strategies.
  • Use the cumulative-volume page to monitor trend bias and configure futures instruments.
  • Start Strategy 1 or Strategy 2, monitor their status, and inspect history snapshots.
  • Open the Manual Trading page, select NIFTY or SENSEX, pick a live-valid strike, and place a manual order.
  • Review positions, orders, logs, and stored trade-history files for analysis and traceability.

Backend Lifecycle

The FastAPI application is defined in app/server.py. On startup it creates a background task that periodically checks Strategy 1 and Strategy 2 during market hours. This is important because their monitoring logic does not depend on the frontend remaining open.

  • Background monitor interval: 10 seconds.
  • Monitor window: 09:15 to 15:30.
  • Monitors active strategy states such as ORDER_PLACED and POSITION_OPEN.
  • Serves the production frontend from frontend/dist when that directory exists.

Backend

Backend Modules and Responsibilities

app/server.py

Creates the FastAPI application, mounts all route groups, handles CORS, exposes a WebSocket endpoint, and optionally serves the built frontend.

config/settings.py

Loads environment variables, creates token/data/log directories, and defines runtime flags such as paper mode, live trading enablement, and risk defaults.

core/auth.py

Handles Zerodha authentication, JWT token generation for manual trading, and token persistence in data/tokens/access_token.json.

core/broker.py

Abstracts order placement, market data, positions, holdings, and order modification/cancellation. Supports exchanges NSE, BSE, NFO, BFO, and MCX.

core/risk_manager.py

Enforces daily loss, trade count, and position sizing constraints before strategies are allowed to continue trading.

engine/trading_engine.py

Loads configured strategies, resolves instrument tokens, subscribes to Zerodha ticks, distributes market data, and handles end-of-day square-off.

Strategies

Automated Strategies and Trading Logic

CumulativeVolumeStrategy

This module is the analytical foundation for the system. It fetches 1-minute OHLCV futures data, computes signed volume, then accumulates that signed volume from market open to derive a directional bias.

  • Uses live data when Zerodha is authenticated.
  • Falls back to realistic demo data when broker auth is unavailable.
  • Exposes a threshold-based trend classification: Bullish, Bearish, or Neutral.
  • Can auto-resolve the current month futures token from the instrument symbol.

Trading Engine

The engine runs configured strategies from ACTIVE_STRATEGIES, subscribes to Zerodha ticks through KiteTicker, and routes ticks to each running strategy during market hours.

  • Market hours: 09:15 to 15:30.
  • Auto square-off trigger: 15:15.
  • Stops feeding ticks if the risk manager disallows trading.

Implemented Strategy Modules

Strategy Type Core Idea Status in Registry
strategy1_gann_cv Option buying Uses cumulative volume direction to buy ATM CE or PE near Gann floor levels with SL and target management. Registered in strategies/registry.py.
strategy2_option_sell Option selling Uses cumulative volume direction to sell ATM options near Gann ceiling levels with inverted premium-decay logic. Implemented as a module and route-backed feature, but not currently present in the central registry map.
cumulative_volume Analysis Non-execution signal generation and chart-oriented market bias analysis. Used by dashboard and strategy pages.
Manual Trading

Manual Trading Module

Manual Auth Model

Manual trading uses JWT-based authentication that is separate from Zerodha login. The frontend currently auto-authenticates using the built-in manual user account: 

username: admin
password: adminpass

The generated bearer token is stored in browser local storage and reused for calls under /api/manual. If a request returns 401, the frontend clears the stored token and retries login once.

Supported Indices

  • NIFTY on exchange NFO
  • SENSEX on exchange BFO

The backend does not rely on fabricated symbols. Instead, it resolves actual live-valid option contracts from the broker instrument dump using index name, option type, strike, and the nearest valid non-expired expiry.

Order Entry Workflow

  1. User opens the Manual Trading page.
  2. Frontend ensures a manual JWT exists, logging in automatically if needed.
  3. Frontend calls GET /api/manual/option_setup for the selected index and option type.
  4. Backend returns the nearest expiry, current spot price, ATM strike, and a strike ladder centered around ATM.
  5. User selects CE or PE, strike, quantity, order type, product, and risk-field values.
  6. Frontend posts the order request to POST /api/manual/order.
  7. Backend resolves the actual tradingsymbol, splits quantity for iceberg legs when needed, and places broker or paper orders.

Current Scope

  • Open positions view
  • Open orders and order history view
  • Trade log and P&L endpoints
  • Order modify and cancel endpoints
  • Square-off action
  • Debug console section in the frontend
Important limitation: the manual order form captures stop loss, target, trailing stop, move-to-cost, and re-entry fields, but the current backend does not yet implement a full live automated monitoring engine for manual trades equivalent to the dedicated strategy modules. Those fields are presently input parameters rather than a complete manual-trade lifecycle automation system.

Security and Controls

Authentication, Risk, and Safety Features

Authentication Layers

  • Zerodha session authentication for market data and real broker connectivity.
  • Manual JWT authentication for /api/manual endpoints.
  • Manual auth user store is currently hard-coded in core/auth.py.

Safety Features

  • Paper trade mode is on by default.
  • Live mode requires both broker auth and explicit enablement via TRADING_ENABLED=true.
  • Daily loss limit and max trades per day are configurable.
  • End-of-day square-off is implemented in the engine and strategy monitoring paths.
  • Position and order sizing are governed by risk defaults in configuration.
Security note: the manual JWT secret and default manual user credentials are currently embedded in application code. For production use, these should be externalized to environment variables or a secure secrets store and the default credential pair should be replaced.
Operations

Operational Notes, Known Constraints, and Recommendations

Logging

Application logs are written under logs/. Strategy trade history and manual trade records are also stored in data and CSV files for later review.

Port Consistency

Current code uses a dev-proxy split: frontend on 5173 and backend target on 8010, while legacy docs and CLI startup still refer to 8000.

Manual SL/Target

Manual order risk fields exist in the UI and request model, but full automated manual-trade lifecycle enforcement is not yet implemented.

Recommended Next Improvements

  • Unify documentation, CLI defaults, and Vite proxy so backend port usage is consistent.
  • Move manual auth secrets and credentials out of source code.
  • Implement a dedicated monitoring engine for manual SL, target, trailing stop, and re-entry logic.
  • Expand the central strategy registry if Strategy 2 should be runnable through the generic CLI engine.
  • Add deployment-specific documentation once a production environment layout is finalized.
